# Data Model: User Sessions Management

## Entities

### PersonalAccessToken

This entity represents a Laravel Sanctum session token stored in the database.

* **Database Table**: `personal_access_tokens`

#### Fields

| Field Name | Type | Key | Nullable | Description |
|------------|------|-----|----------|-------------|
| `id` | bigint(20) unsigned | PK | No | Unique identifier for the token. |
| `tokenable_type` | varchar(255) | | No | Morph type (always `App\Models\User`). |
| `tokenable_id` | bigint(20) unsigned | FK/Index | No | ID of the User associated with this token. |
| `name` | varchar(255) | | No | Device name or User-Agent identifying the session. |
| `token` | varchar(64) | Unique | No | SHA-256 hash of the plain-text API token. |
| `abilities` | text | | Yes | JSON or serialized capabilities (typically `["*"]`). |
| `last_used_at` | timestamp | | Yes | The date and time this token was last used. |
| `expires_at` | timestamp | | Yes | The date and time this token expires. |
| `created_at` | timestamp | | Yes | Creation timestamp (Session start time). |
| `updated_at` | timestamp | | Yes | Last update timestamp. |

#### Relationships

* `User` (1-to-Many): A `User` has many `PersonalAccessToken` records.
  - Defined via `$user->tokens()` relationship in Laravel.
  - Mapped inversely on the model as `tokenable`.

#### Validation Rules

Since the token list is generated by Sanctum's login mechanism and can only be deleted, no creation validation rules are required for this entity in the profile context. 

For the deletion/revocation action:
- **`id` (Route Parameter)**: MUST be an integer matching an existing token ID owned by the authenticated user.
- **Auto-exclusion Constraint**: The token `id` MUST NOT equal `$request->user()->currentAccessToken()->id`. If it does, a `400 Bad Request` is returned.
- **Ownership Constraint**: The token's `tokenable_id` MUST match the authenticated user's ID. If not found or owned by someone else, a `404 Not Found` is returned.
